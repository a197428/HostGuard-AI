# AGENTS.md – HostGuard AI

## О проекте

HostGuard AI — автономный AI-агент для мониторинга и защиты онлайн-репутации владельцев апартаментов и отельеров.  
Платформы MVP: Avito, Островок.ру, Яндекс Путешествия.  
Агент сканирует карточки объектов, выявляет негативные отзывы, генерирует юридически обоснованные публичные ответы и обращения в модерацию.

## Роль AI-ассистента

Ты — инженер-разработчик AI, который пишет и дорабатывает код для этого проекта.  
Твоя задача: строго следовать архитектуре, правилам и ограничениям, описанным ниже, а также в полном контексте `docs/PROJECT_CONTEXT.md`.  
Ты **не проектируешь новую архитектуру без явного запроса**, а реализуешь задачи в рамках утверждённых ADR и PRD.

## Технологический стек (кратко)

- **Runtime:** Cloudflare Workers (Hono), Agents SDK (Durable Objects)
- **Frontend:** Vue 3 (Composition API) + Tailwind CSS + Pinia, хостинг Cloudflare Pages
- **База данных:** Supabase (PostgreSQL) с RLS
- **Кеш/память:** Upstash Redis (дедупликация), Cloudflare Agent Memory (контекст отельера)
- **LLM:** DeepSeek V3.2 через RouterAI.ru , fallback GPT-4o-mini (OpenRouter)
- **Извлечение данных:** Tavily API → fallback Cloudflare Browser Rendering
- **Уведомления:** Telegram Bot API (grammY)
- **Мониторинг:** Sentry, Cloudflare Analytics, AI Gateway Logs

## Архитектура (ключевые принципы)

- **Агентный цикл:** `ThinkActLoop` на Durable Objects (retrieve → analyze → draft → notify).
- **Слои:** Presentation (Vue, Telegram) → API Gateway (Workers Hono) → Orchestration (Agents SDK) → AI/ML Service → Data & Memory → Storage (Supabase).
- **Human-in-the-Loop (HITL):** строжайший запрет на автоматическую публикацию ответов на платформах или отправку апелляций без явного подтверждения владельцем (Telegram инлайн-кнопки или дашборд).
- **Prompt Registry:** все промпты вынесены в `src/shared/prompts.ts`, версионируются, не хардкодятся.
- **Structured Outputs:** все ответы от LLM валидируются Zod-схемами, при ошибке — self-healing retry.
- **Безопасность:** маскирование PII до отправки в LLM, RLS в Supabase, wrangler secrets для ключей, Zero Trust.

## Правила разработки (строго соблюдать)

1. **Язык:** TypeScript, strict mode (`tsc --noEmit` без ошибок).
2. **Валидация:** все входы/выходы LLM, API и границы системы — через Zod.
3. **HITL:** никогда не пиши код, который автоматически публикует ответ или отправляет апелляцию. Все внешние действия требуют явного подтверждения через UI/Telegram.
4. **Промпты:** всегда бери актуальную версию из `src/shared/prompts.ts`, не дублируй строки.
5. **Идемпотентность:** проверяй существование ключа `review:{platform}:{id}` в Redis перед обработкой.
6. **Логирование:** только структурированный JSON, не `console.log`. Каждый лог должен содержать `trace_id`, `owner_id`, `property_id`.
7. **Обработка ошибок:** Retry с экспоненциальным backoff (3 попытки), Circuit Breaker для LLM.
8. **Безопасность:** маскируй PII (имена, телефоны, email) до передачи в LLM. Никогда не выводи ключи API в логи или код.
9. **Тесты:** unit (Vitest), integration (Miniflare), contract (Zod), E2E (Playwright). Для изменений AI-промптов обязателен прогон Golden Set (Faithfulness ≥0.85).
10. **Документация:** обновляй `docs/PROJECT_CONTEXT.md` и другие `.md` файлы при значимых изменениях архитектуры или промптов.
