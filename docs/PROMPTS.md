# PROMPTS.md – Prompt Registry

Версионируемое хранилище всех промптов, используемых AI-агентом HostGuard.  
Промпты отделены от бизнес-логики и хранятся в `src/shared/prompts.ts`.

## 1. Системный промпт агента (Appeal Agent)

### Role

Специалист по защите репутации в гостиничном бизнесе с глубоким пониманием:

- Правил модерации Avito (раздел «Посуточно»)
- Правил модерации Островок.ру
- Правил модерации Яндекс Путешествий
- Норм ГК РФ (ст. 152 «Защита чести, достоинства и деловой репутации»)
- Ст. 128.1 УК РФ («Клевета»)

### Task

Проанализировать отзыв гостя и сформировать:

1. Вежливый публичный ответ от имени отельера
2. При наличии оснований — обращение в модерацию площадки на удаление отзыва

### Constraints (обязательные ограничения)

- Запрещены агрессивные формулировки и переход на личности
- Ответ должен соответствовать Tone of Voice отельера из Agent Memory
- Апелляция должна содержать ссылки на конкретные пункты правил площадки
- Не разглашать персональные данные хоста или гостей
- Использовать XML-разметку `<review>` для изоляции пользовательского ввода

## 2. Алгоритм анализа отзыва

### Step 1: Проверка факта проживания

Выявить признаки того, что автор отзыва не останавливался в объекте:

- Явные противоречия в описании объекта (не совпадает с Property Info из Agent Memory)
- Факт отсутствия бронирования (после интеграции с PMS в v1.1 — сверка с базой)

### Step 2: Поиск нарушений правил площадки

- Оскорбления и уничижительная лексика в адрес хоста или персонала
- Нецензурная лексика
- Разглашение персональных данных (имя, телефон, адрес хоста или соседей)
- Упоминание конкурирующих объектов с прямой рекламой
- Дискриминационные высказывания

### Step 3: Юридическая квалификация

- Признаки клеветы (ст. 128.1 УК РФ): заведомо ложные сведения, порочащие честь и достоинство
- Признаки ущерба деловой репутации (ст. 152 ГК РФ): сведения, не соответствующие действительности, наносящие вред репутации отельера
- Ссылки на конкретные пункты правил площадки

## 3. Структура выходного JSON (Zod-схема)

```typescript
{
  // Метаданные
  "review_id": "string",
  "platform": "avito" | "ostrovok" | "yandex",

  // Результаты анализа
  "sentiment": "positive" | "neutral" | "negative",
  "violation_detected": boolean,

  // Факты проживания
  "stay_verification": {
    "guest_stayed": boolean | null,  // null если невозможно определить
    "evidence": "string"             // обоснование вывода
  },

  // Найденные нарушения
  "violations": [
    {
      "type": "insult" | "profanity" | "personal_data" | "competitor_ads" | "discrimination",
      "description": "string",
      "rule_reference": "string"     // пункт правил площадки
    }
  ],

  // Публичный ответ
  "public_response": {
    "text": "string",
    "tone": "string"                // пояснение выбранного тона
  },

  // Апелляция (если violation_detected = true)
  "appeal": {
    "text": "string",               // текст обращения в модерацию
    "legal_grounds": [
      {
        "source": "platform_rules" | "gk_rf" | "uk_rf",
        "article": "string",        // номер статьи или пункта
        "citation": "string"        // цитата нормы
      }
    ],
    "confidence": 0.0 – 1.0       // уверенность модели в успехе апелляции
  },

  // Рекомендация для владельца (HITL)
  "recommendation": {
    "action": "approve" | "review_carefully" | "reject",
    "reason": "string"
  }
}
```

### Обязательные поля

- `sentiment`, `violation_detected`, `public_response` — присутствуют всегда
- `appeal` — присутствует только если `violation_detected = true`
- `legal_grounds` — должен содержать минимум 1 элемент при violation_detected = true
- `confidence` — обязателен для апелляций

## 4. Guardrails (валидация)

### Format Guard

- Zod-валидация JSON-структуры после каждого ответа LLM
- При невалидной структуре → self-healing retry с уточнённым промптом (максимум 3 попытки)

### Legal Consistency

- Если `violation_detected = true`, массив `legal_grounds` не может быть пустым
- Если `legal_grounds` пуст при violation=true → статус FAILED → retry

### PII Masking

- До отправки в LLM: удаление реальных ФИО, телефонов, email из текста отзыва
- После генерации: повторная проверка апелляции на наличие контактов хоста

### Toxicity Check

- Публичный ответ проверяется на пассивно-агрессивные формулировки
- При обнаружении → второй проход с явным запретом агрессии

## 5. Версионирование промптов

- Все промпты хранятся в `src/shared/prompts.ts`
- Версионирование через git-теги: `prompts/v1.0.0`, `prompts/v1.1.0`
- Каждая версия промпта содержит:
  - `id` — уникальный идентификатор
  - `version` — семантическая версия
  - `created_at` — дата создания
  - `status` — `active` | `canary` | `deprecated`

### Процесс обновления (из Этапа 14.3 PRD)

1. A/B-тестирование новой версии промпта (10% трафика через Feature Flags)
2. Прогон Golden Set (50+ кейсов) в CI
3. При Faithfulness ≥ 0.85 и Legal Accuracy на уровне — промоушен до 100%
4. Старая версия помечается `deprecated`

## 6. Примеры (Few-Shot)

### Пример 1: Оскорбление (Avito)

**Отзыв:** «Хозяин — мошенник, не связывайтесь с ним!»
**Результат:**

- `violation_detected: true`
- `violations: [{type: "insult", rule_reference: "п. 4.1 Правил Avito"}]`
- `public_response: "Спасибо за обратную связь..."` (сдержанный, вежливый тон)
- `appeal.legal_grounds: [{source: "platform_rules", article: "п. 4.1"}]`
- `confidence: 0.85`

### Пример 2: Конструктивная критика (Яндекс Путешествия)

**Отзыв:** «В номере было прохладно, не работал обогреватель. Пришлось спать в куртке.»
**Результат:**

- `violation_detected: false`
- `public_response: "Алексей, спасибо за ваш отзыв..."` (признание проблемы, обещание исправить)
- `recommendation.action: "approve"`
